# LabVIEW P&ID 动态组态设计器 (WPF 混合架构) 设计方案

## 1. 核心定位与架构总览
本软件致力于解决“不同设备 P&ID 图需要重新硬编码绘制”的痛点，旨在打造一个专业、顺滑且可高度复用的动态组态平台。系统采用强解耦的 **WPF 独立设计器 + 编译型 .NET 控件** 的架构。

整体工作流如下：
1. **设计阶段**：工程师在脱离 LabVIEW 的 **WPF 独立设计器 (EXE)** 中通过拖拽完成 P&ID 组态设计。
2. **导出阶段**：设计器将画布状态序列化为 `.xml` 文件（用于二次编辑），并**直接生成/编译出一个原生的 .NET 控件 DLL**（如 `CustomPID.dll`），其内部已固化了排版与渲染逻辑。
3. **运行阶段**：LabVIEW 仅需将该 DLL 放置在前面板的 .NET 容器中，无需在运行时解析 XML。LabVIEW 作为纯数据引擎，通过通用 API 高频注入数据即可。

---

## 2. 独立组态设计器 (Designer EXE)
这是一个纯 WPF 构建的桌面应用程序，专为工程师提供流畅的绘图体验。

*   **图元组件库 (Component Library)**：内置高度封装的 WPF `UserControl` 图元。包括：管道、阀门、泵、加热器、板换、水罐、MFC、温度计、压力计、氢浓度传感器、流量计等。
*   **网格与吸附系统**：支持背景网格显示，图元拖拽时自动吸附对齐。
*   **正交自动布线 (Orthogonal Routing)**：采用基于图元锚点（Anchors）的智能寻路算法。点击起点和终点后，系统自动避开障碍图元，生成带有直角转弯的管道连接。
*   **属性配置器**：选中图元后，可在右侧面板配置其专有属性，最核心的是为其分配全局唯一的绑定标签（如 `ID="Pump_Main"`）。
*   **双轨导出机制**：
    *   **保存工程**：导出 `.xml`，保留完整的对象拓扑关系，随时可导入 EXE 进行“二次创作”。
    *   **生成 DLL**：通过内置的编译器或代码生成机制，将当前画布硬编码转换为一个独立的 `.NET DLL`，专供 LabVIEW 调用。

---

## 3. 运行时控件与数据契约 (Runtime DLL & Data Flow)
导出的 `.NET DLL` 具备极高的运行效率和极简的外部接口。

### 核心数据契约 (Data Contract)
为了应对未来不可预知的图元扩展，LabVIEW 与控件之间采用**扁平化通用簇数组 (Flat Cluster Array)** 作为唯一的数据总线。

**LabVIEW 簇结构定义：**
*   `ID` (String): 唯一标识符（必须与设计器中配置的 ID 严格一致）
*   `Type` (Int32): 图元类型枚举（1=泵, 2=阀门, 3=温度计...）
*   `Value` (Double): 传感器的实时读数或设备的设定值
*   `State` (Int32): 状态机枚举（0=停止/关闭, 1=运行/打开, 2=报警/故障...）

### API 交互模式
*   **下行 (LabVIEW -> UI)**：
    DLL 开放接口 `UpdateDeviceStates(Cluster[])`。LabVIEW 在后台通过 `.NET Invoke Node` 定时循环调用此接口。DLL 接收到数组后，通过比对 `ID`，将对应的 `Value` 和 `State` 路由到具体的底层 WPF 图元，触发数值更新或复杂的 XAML 动画（如泵的叶轮旋转、管道颜色流动）。
*   **上行 (UI -> LabVIEW)**：
    DLL 暴露 `.NET Event`。当操作员在界面上点击某个支持反向控制的图元（如切换阀门状态）时，DLL 会抛出事件并附带 `ID` 和指令，LabVIEW 的事件结构捕获该动作并下发至真实硬件。

---

## 4. 方案优势总结 (Critical Evaluation)
1. **极致的运行时性能**：剥离了沉重的运行时 XML 解析，导出的就是原生渲染的 .NET 控件，UI 线程压力极小。
2. **零成本增量维护**：LabVIEW 代码逻辑极为清晰，底层数据结构（簇数组）和通信接口永远固定。即便未来新增了新型传感器或复杂设备，也只需在 WPF 端增加图元模板，LabVIEW 侧无需任何改动。
3. **所见即所得**：依托 WPF 强大的硬件加速和 XAML 动画系统，P&ID 界面的视觉保真度和科技感将远超传统 LabVIEW 原生界面。
